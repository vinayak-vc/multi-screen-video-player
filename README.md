# Multi-Screen Video Player

## 1. Project Overview & Architecture
The **Multi-Screen Video Player** is a networked Unity application designed to playback and synchronize video content across multiple displays (or a stereoscopic player), remotely controlled by an Android device.

The architecture strictly follows a **Client-Server (Host-Remote)** model using **Unity Netcode for GameObjects**. Commands (Play, Pause, Stop, Browse) generated by the UI on the remote Android client are serialized into strings (via the `Commands` class) and dispatched to the Windows Host using Netcode RPCs. The Windows Host executes these commands against local `VideoPlayer` components mapped to various displays, and continuously streams playback progress and state back to the Android client.

### Key Architectural Pillars:
1. **Network Layer:** Standard Unity Netcode (`UnityTransport`) dynamically configuring itself as Host (if on Windows) or Client (if on Android). 
2. **Video Rendering:** Unity's native `VideoPlayer` routing output to `RenderTexture` instances. Multiple Cameras isolate displays (e.g., Target Display 1, 2, 3), scaling out linearly with connected physical monitors.
3. **Stereoscopic Integration:** An alternative WebSocket-based pipeline (`BasicWebSocketClient`, `StereoscopicComController`) allows the Windows Host to control external stereoscopic video software instead of local Unity UI Canvas elements.
4. **360 Degree Video:** Supports playback of 360-degree spherical videos. Includes an interactive Android touchpad interface to capture and send camera panning data to the Windows Host.

## 2. Modules & Script Responsibilities

### Core Networking Module
* **`BootstrapManager.cs`**: The entry point for the network. It instantiates the NetworkManager and configures the IP/Port. It determines via the `isThisWindows` boolean whether to start a Host (Windows) or a Client (Android). 
* **`NetworkMediator.cs`**: A `NetworkBehaviour` inheriting script that acts as the messaging bridge. It implements `ServerRpc` and `ClientRpc` to transmit string commands between the Android client and the Windows host.
* **`Commands.cs`**: A static constants class that defines the unified vocabulary of network commands (e.g., `Play`, `Pause`, `SliderData`, `GetImages`, `Separator`).

### Windows Host Module (Video & Display)
* **`WindowsPlayer.cs`**: The core controller on the Host PC. It detects connected displays (`Display.displays[i].Activate()`) and instantiates a `VideoPlayerController` for each video in the playlist. It parses incoming network commands and proxies them to the correct video player, or forwards them to an external stereoscopic app via WebSockets.
* **`VideoPlayerController.cs`**: Wraps Unity's native `VideoPlayer`. It allocates the dynamically created `RenderTexture`, spawns UI Canvases assigned to specific target displays, and exposes standard media controls (Play, Pause, Seek, Mute) that the `WindowsPlayer` invokes.
* **`WindowsUIController.cs`**: Manages the host's debugging UI (such as IP address display) and folder picking interface if interactions are executed directly on the host rather than remotely.

### Android Client Module (Remote Controller)
* **`AndroidPlayer.cs`**: The client-side counterpart to `WindowsPlayer`. It listens for messages from the host (such as `SliderData` or `NameVideo`) and translates them into UI updates through the `MobileUIController`.
* **`MobileUIController.cs`**: Dominates the Android UI layer. It manages the connection screen, highlights the currently playing video in the playlist, sets the seek-bar progress seamlessly, and captures remote inputs to dispatch to the network.
* **`ImmersiveManager.cs`**: A specialized UI manager utilizing fading animations (`FadeAnimationCanvasGroup`) handles transitions between the playlist view and the full-screen playback controls.
* **Touchpad Controls**: New scripts and interfaces capturing user swipe/delta inputs on the mobile device, utilizing the `Commands.InputCommands.Delta` structure to remotely rotate the host's 360-degree view.
* **`SliderPointerEvents.cs`**, **`ButtonController.cs`**, **`HoverInfo.cs`**: Modular interaction scripts attached to UI components to handle scrolling, hovering, and specialized button triggers.

### Stereoscopic Support Module
* **`BasicWebSocketClient.cs`**: Allows the Windows Host to open a secondary websocket connection alongside the Unity Netcode connection.
* **`StereoscopicComController.cs`**: Uses the websocket to send specialized commands directly to third-party stereoscopic players (like Bino) instead of relying on Unity's UI Canvases.

### Discovery & File Management Module
* **`FileExplorer.cs`**, **`FolderObjects.cs`**, **`PathInfoObject.cs`**: Utility scripts used to scan local or network paths for valid `.mp4` payloads and thumbnails, populating the UI lists dynamically.

## 3. Scenes & Their Roles

### `Bootstrap/Scenes/Windows.unity`
* **Role**: The Server/Host executable scene.
* **Usage**: Compiled exclusively for the Windows PC wired to the multiple displays. It contains the `BootstrapManager` set to `isThisWindows = true` and `WindowsPlayer` components. This scene remains mostly "invisible" locally, routing standard UI to secondary screens while waiting for network commands.

### `Bootstrap/Scenes/Android.unity`
* **Role**: The Client Remote App scene.
* **Usage**: Compiled for Android tablets or phones. Contains the IP entry screen and the remote control interface (`MobileUIController`, `AndroidPlayer`). The user carries this device to manipulate the Windows Host.

### `Bootstrap/Scenes/Windows 360.unity`
* **Role**: The Host executable scene specialized for 360-degree interactive playback.
* **Usage**: Extends the core `WindowsPlayer` framework to render spherical content onto a dedicated `360Sphere` prefab. Processes incoming mobile Touchpad delta commands to manipulate and rotate the main camera view.

### `Bootstrap/Scenes/Mix.unity`
* **Role**: Unified Testing/Standalone scene.
* **Usage**: Used locally in the Unity Editor or for a single-screen standalone deployment. It blends both Host and Client modules so developers can test the networking and UI flow without needing two discrete physical devices constantly connected.
